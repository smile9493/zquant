# Workspace 状态规范

## 目标

保证桌面工作台“可恢复、可追踪、可回放”，避免状态散落与并发冲突。

## 状态分层

- `UI State`：面板开关、焦点、临时选择。
- `Workspace State`：symbol、timeframe、视图范围、最近操作上下文。
- `Domain State`：数据集、任务状态、同步状态。
- `Render State`：供渲染使用的不可变快照。
- `Domain State` 与 `Render State` 必须严格分治，禁止混存。

## 业务真状态 vs 渲染派生状态

业务真状态（归 `core/domain`）：
- 当前项目、策略参数、数据源配置
- 任务生命周期、回测条件、工作区持久化布局

渲染派生状态（归 `renderer-bevy`）：
- 相机位置、GPU 高亮、帧统计
- hover/picking 临时结果、离屏纹理句柄、动画插值中间态

禁止事项：
- 将业务真状态挂入 ECS Resource 作为权威来源
- 由渲染层直接改写 `Workspace State` 或 `Domain State`

## 演进模型

采用 `Command -> Reducer -> Snapshot -> Event`：

- `Command`：意图输入（用户动作或系统触发）。
- `Reducer`：纯状态演算，最小化副作用。
- `Snapshot`：可序列化的稳定状态。
- `Event`：用于通知、日志、任务面板同步。

## 持久化与恢复

- Workspace 快照至少包含：symbol/timeframe、视图窗口、面板状态、最近打开对象。
- 应用启动时按“上次有效快照”恢复；快照损坏时降级到默认布局。
- 状态版本号必须可升级（schema version），禁止隐式破坏兼容。

## 并发与一致性

- 同一工作区的状态写入必须串行化或版本化（乐观并发）。
- 后台任务更新状态时必须带来源与时间戳。
- 任务结果回写不得直接覆盖用户交互中的临时状态。
- 输入路由冲突结果应作为事件进入 `Reducer`，禁止旁路改状态。

## 质量门禁

- 异常退出后可恢复到最近稳定快照。
- 工作区切换不会污染其他工作区状态。
- 关键状态变更可在日志/事件面板追踪。
- 核对 `core/domain` 不依赖 UI/渲染框架，保证可被 CLI/测试程序独立驱动。
