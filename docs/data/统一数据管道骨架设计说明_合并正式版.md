## 2. 执行摘要

### 2.1 建设背景概述

当前项目已经完成 Phase 1 单机 EDA 内核，具备 PostgreSQL 作为状态真相源、单进程 kernel、in-memory event bus、Job lifecycle 事件链、Agent 编排闭环以及基础可观测与回归验证能力。在这一基座之上，系统已经具备承载“数据拉取 → 数据清洗 → 质量门控 → 持久化写入 → 事件发射”主链路的控制面与执行面基础。

与此同时，现阶段的数据接入能力仍然缺少一个统一入口。若上游直接绑定具体 Provider，将导致调用方式碎片化、路由策略分散、数据标准不一致、质量门控位置不统一，并使后续扩展缓存、Kafka、健康探针、更多 Provider 或前端控制台能力时产生过高耦合。

因此，本次建设的重点不是复刻旧系统的全部运行形态，而是抽取统一数据管道中稳定、可迁移、可扩展的骨架能力，形成一套适用于新项目的新方案设计蓝图。

### 2.2 本次建设目标

本次建设的直接目标是形成面向新项目的统一数据管道骨架，使调用方不再直接依赖具体 Provider，而是通过统一入口，以能力、市场、数据集契约驱动请求，由统一管道负责 Provider 选择、路由与降级、标准化、质量门控、持久化写入、事件发射与可观测埋点。

### 2.3 本次交付范围

本次交付聚焦骨架层与最小闭环，包括：

- 统一入口 `DataPipelineManager`
- Provider 统一契约
- Capability / Market / Dataset 契约
- 最小路由责任链
- 标准化处理
- DQ 主链路门控
- 持久化与 quarantine 机制
- 统一事件发射接口
- 与现有 Job / Agent / EventBus / 可观测体系的对接点

### 2.4 本次不包含范围

当前阶段明确不包含以下全量能力：

- Redis 两级缓存完整体系
- Kafka 全链路前置接入
- 配额感知路由完整系统
- 后台健康探针调度器完整实现
- 前端配额管理 UI
- 全量 Provider 生态迁移
- 旧系统大规模兼容层整体搬迁
- 旧注册表或旧部署形态的原样复制

### 2.5 预期业务与技术收益

建设完成后，系统将获得以下收益：

- 数据接入入口统一，减少上游散乱调用
- Provider 切换、降级与扩展变为内部能力
- 数据标准化与质量门控口径统一
- reject / degraded / accept 三态治理清晰
- 新项目与 Job / Agent / EDA 的协同关系稳定
- 为 Redis、Kafka、健康探针、更多 Provider 和前端工作台提供平滑扩展点

## 3. 背景与问题定义

### 3.1 现状背景

当前新项目已经完成控制面与执行面的第一阶段建设，具备处理异步任务、事件驱动流转以及基础观测的核心骨架。此时开始建设统一数据管道，具备明确的工程时机：上层调用需求开始出现，而底层执行能力也已经具备承接主链路的条件。

### 3.2 当前主要问题

当前若继续沿用分散式接入方式，将面临以下问题：

1. **入口不统一**：上游可能直接绑定底层 Provider SDK 或第三方接口。
2. **路由逻辑散落**：能力匹配、市场选择、Provider 强制指定、失败降级可能在多个位置重复实现。
3. **标准不一致**：字段命名、时间语义、数值类型和元数据补齐规则难以统一。
4. **质量门控不稳定**：DQ 可能被当作后处理插件，而不是强制进入主链路。
5. **持久化与审计分裂**：数据落盘、元数据写入、事件发射无法形成稳定闭环。
6. **后续扩展成本高**：缓存、Kafka、健康探针、观测、更多 Provider 接入会反复改动上游代码。

### 3.3 现有系统约束

当前系统存在以下明确约束：

- 必须以新项目架构为中心，而不是旧系统迁移为中心。
- 当前阶段仅适合建设骨架与最小闭环，不适合一次性引入全部外围能力。
- 调用方应当只声明需求，而不承担 Provider 差异处理责任。
- 数据质量能力必须进入 ingest 主链路。
- 基础设施抽象必须可替换，允许后续平滑接入 Redis / Kafka / 分布式执行。

### 3.4 建设必要性

若不在当前阶段建立统一数据管道骨架，未来随着 Job、Agent、脚本、API 等调用方增加，系统将迅速形成多个“私有接入通道”，导致后续治理成本显著上升。相反，在当前基座稳定后尽快收敛统一入口，可以在成本最低的阶段锁定上层契约和下层扩展边界。

## 4. 建设目标与非目标

### 4.1 建设目标

#### 4.1.1 建立统一数据接入入口

所有外部数据接入必须通过统一入口组件进行，禁止上游直接调用底层 Provider SDK 或第三方 API。

#### 4.1.2 建立统一 Provider 契约与能力模型

所有 Provider 通过统一契约接入，并以标准化 Capability、Market、Dataset 协议表达能力边界。

#### 4.1.3 建立统一标准化、质量门控、入库与事件发射链路

统一数据管道必须负责标准化、DQ 决策、持久化、quarantine、元数据写入与事件发射，形成稳定闭环。

#### 4.1.4 建立可观测、可审计、可扩展的数据管道骨架

系统必须对外暴露稳定契约，对内预留缓存、消息总线、健康检查和更多 Provider 的平滑替换能力。

### 4.2 非目标

当前阶段不做以下建设：

- 不引入 Redis 多级缓存作为骨架前置依赖
- 不要求 Kafka 成为主链路必经组件
- 不建设前端配额管理与运营控制台
- 不做旧系统的全量兼容迁移
- 不建设完整 Provider 生态与历史注册表复制
- 不把健康探针、熔断、配额惩罚做成强制主链路能力

## 5. 适用范围与设计边界

### 5.1 适用范围

统一数据管道骨架在当前阶段适用于以下领域：

- 数据拉取
- 数据标准化
- 数据质量门控（DQ）
- 数据持久化
- 元数据与目录写入
- 事件发射
- 基础观测与审计

### 5.2 不适用范围

当前阶段不直接覆盖以下内容：

- 前端展示层实现
- 运营配额平台
- 全量 Provider 生态治理
- 非本阶段目标的实时推送与复杂订阅编排
- 与旧项目完全等价的历史系统形态

### 5.3 设计边界

#### 5.3.1 控制面边界

控制面负责：

- 任务编排

- 可观测与审计

#### 5.3.2 数据面边界

数据面负责：

- 原始数据拉取
- 数据清洗
- 标准化
- 质量门控
- 清洗后数据持久化
- quarantine 写入

#### 5.3.3 当前阶段能力边界

当前阶段重点保证同步与异步入口存在、ingest 主链路闭环存在、reject / degraded / accept 三态存在、对接 Job / Agent / EDA 的边界存在。其余增强能力均以后续阶段按需启用。

## 6. 术语、缩略语与统一定义

- **Provider**：外部数据源或数据服务适配器实现。
- **Capability**：标准化能力标签，用于表达 Provider 可提供的数据能力。
- **Dataset**：可被统一拉取和摄入的数据集对象。
- **Ingest**：从数据拉取到质量门控再到持久化落盘的完整摄入过程。
- **DQ**：Data Quality，数据质量治理与门控能力。
- **PiT**：Point in Time，面向时间语义的数据落盘与可追溯要求预留位。
- **SSOT**：Single Source of Truth，唯一真相源。
- **Quarantine**：隔离区，用于存放 reject 数据及其质量元信息。
- **Fail-Closed**：当强约束不满足时显式失败，而非静默换源成功。
- **Fallback**：在允许的策略范围内，尝试降级到后续候选 Provider。
- **Event Bus**：统一事件通道，用于领域事件发布和下游消费。

## 7. 总体架构设计

### 7.1 架构定位

统一数据管道不是独立平行系统，而是新项目控制面与数据面的中间骨架层。其核心定位是：对上提供统一数据接入与摄入能力，对下封装 Provider 差异和数据治理主链路。

### 7.2 总体分层

系统建议按以下六层组织：

1. **调用层**：Jobs / Agents / Scripts / API
2. **管理层**：DataPipelineManager 统一入口
3. **Provider 层**：各类 market / reference / dataset / local/mock provider
4. **标准化与 DQ 层**：Normalizer、DQGate
5. **存储与事件层**：PersistWriter、CatalogWriter、EventEmitter、QuarantineWriter
6. **可观测层**：metrics、tracing、structured logs

### 7.3 总体架构图

```text
Callers (Jobs / Agents / Scripts / API)
                │
                ▼
      DataPipelineManager（唯一入口）
      ├─ fetch()
      ├─ fetch_dataset()
      ├─ ingest_dataset()
      ├─ fetch_async()
      ├─ fetch_dataset_async()
      └─ ingest_dataset_async()
                │
                ├─ Capability / Market / Dataset Contract
                ├─ Provider Registry
                ├─ Route Resolver
                ├─ Normalizer
                ├─ DQ Gate
                ├─ Persist Writer
                ├─ Event Emitter
                └─ Metrics / Tracing
                │
                ▼
         Provider Adapters
         ├─ Market Data Provider
         ├─ Reference Data Provider
         ├─ Dataset Provider
         └─ Local / Mock Provider
```

### 7.4 核心数据流

系统包含两条核心主链路：

#### 7.4.1 fetch 主链路

```text
caller request
  → resolve providers
  → try fetch
  → normalize
  → return standardized result
```

#### 7.4.2 ingest_dataset 主链路

```text
caller request
  → fetch_dataset
  → normalize
  → DQGate.clean_and_validate
  → decide reject / degraded / accept
  → persist cleaned data
  → write metadata / catalog
  → emit domain events
  → return ingest result
```

### 7.5 与 Phase 1 EDA 内核关系

统一数据管道骨架以现有 Phase 1 EDA 内核作为运行底座，而不是平行复制一套运行时。具体表现为：

- Job 系统可调用 `fetch_dataset` / `ingest_dataset`
- 事件总线负责承接数据相关领域事件
- Agent 层可用于多步骤任务编排
- tracing 与 metrics 复用当前可观测基座

## 8. 核心设计原则

### 8.1 统一入口

所有外部数据接入统一经由 `DataPipelineManager`，上游禁止直接绕过统一入口调用 Provider。

### 8.2 契约优先

调用方只声明需求，包括能力、市场、dataset、Provider 强制约束与质量门控要求，不关心底层重试、路由、降级与标准化细节。

### 8.3 能力驱动路由

系统以能力、市场与数据集契约驱动路由，而不是以具体 SDK 函数名或零散脚本驱动。

### 8.4 DQ 前置于持久化

数据质量门控必须进入 ingest 主链路，在持久化之前完成 reject / degraded / accept 决策。

### 8.5 Fail-Closed 优先于静默降级

当调用方显式指定 Provider 或约束较强时，系统必须优先显式失败，避免无感知换源导致数据语义漂移。

### 8.6 可审计与可追溯

关键动作必须具备日志、事件、质量报告、元数据回写与隔离区写入能力，以支持后续审计与排障。

### 8.7 可替换的基础设施抽象

当前阶段允许采用本机最小实现，但所有核心边界必须支持未来替换为 Redis、Kafka、WS Bridge、分布式 Worker 等实现。

### 8.8 渐进演进而非一次性重构

当前阶段只建设骨架与最小闭环，后续增强能力在不破坏上层接口的前提下逐步演进。

## 9. 核心抽象与接口契约

### 9.1 DataPipelineManager / DataSourceManager

统一入口组件建议暴露以下最小接口：

- `fetch(request)`
- `fetch_dataset(request)`
- `ingest_dataset(request)`
- `fetch_async(request)`
- `fetch_dataset_async(request)`
- `ingest_dataset_async(request)`

其职责包括：

- 接收标准化请求对象
- 基于能力 / 市场 / dataset 解析候选 Provider
- 执行最小路由与失败降级
- 统一归一化输出
- 驱动 DQ 与持久化主链路
- 发射观测与领域事件

### 9.2 Provider 抽象

所有 Provider 必须通过统一契约接入，至少声明：

- `provider_name`
- `capabilities`
- `markets`
- `priority`
- `supports_dataset_ids` 或 dataset resolver
- `fetch()`
- `fetch_dataset()`

可选声明：

- `rate_limit_config`
- `health_check()`
- `supports_async`
- `metadata()`

### 9.3 Capability 模型

首批建议保留以下标准能力集合：

- `OHLCV_DAILY`
- `OHLCV_MINUTE`
- `REFERENCE`
- `MASTER_DATA`
- `DATASET_GENERIC`
- `FUNDAMENTAL`
- `REALTIME`

可预留但不要求本阶段完整实现的能力：

- `TICK`
- `ORDERBOOK`
- `ALTERNATIVE`

### 9.4 Dataset 协议

所有 dataset 通过统一契约描述，最小字段包括：

- `dataset_id`
- `data_type`
- `market`
- `symbol_scope`
- `time_range`
- `provider_hint`
- `quality_gate`
- `persist_policy`

### 9.5 Ingest 协议

`ingest_dataset()` 的返回结构建议至少包含：

- `dataset_id`
- `provider_used`
- `quality_score`
- `decision`（accept / degraded / reject）
- `persisted_path`
- `quarantine_path`
- `issues`
- `metadata_ref`
- `event_ids` 或等价事件标识

### 9.6 事件契约

统一数据管道完成关键动作后应发射统一事件，包括但不限于：

- `dataset.fetched`
- `dataset.gate.completed`
- `dataset.ingested`
- `dq.rejection`
- `dq.degraded`

事件契约应保持字段稳定，以便后续替换底层消息总线而不影响上游与下游语义。

### 9.7 错误模型与返回语义

错误模型建议至少区分：

- 参数错误
- 契约不匹配
- 无可用 Provider
- Provider 拉取失败
- 标准化失败
- DQ reject
- 持久化失败
- 事件发射失败

其中 DQ reject 不应简单视为异常终止，而应是具备明确业务语义的受控返回路径。

## 10. 核心处理流程设计

### 10.1 同步拉取主链路

同步拉取流程应满足以下要求：

1. 接收标准化请求
2. 解析候选 Provider 链
3. 按优先级尝试拉取
4. 对结果进行标准化
5. 返回统一结构对象

### 10.2 数据标准化主链路

所有 Provider 输出必须通过统一标准化层处理，至少完成：

- OHLCV 列名归一
- 时间字段归一
- 数值类型归一
- symbol / market / dataset 元信息补齐

### 10.3 DQ 校验与决策主链路

DQGate 负责调用主质量引擎，产出：

- `cleaned_df`
- `quality_score`
- `issues`
- `should_reject`

同时根据阈值决定：

- 正常入库
- 降级入库
- 拒绝入库并写 quarantine

### 10.4 入库主链路

通过 DQ 决策后，系统负责：

- 持久化清洗后数据
- 写入 catalog / metadata
- 预留 PiT / available_at / SSOT 元数据接口
- 保障幂等写入能力

### 10.5 事件发射主链路

在 fetch、gate、ingest 等关键动作完成后，系统统一发射领域事件，用于：

- 下游消费
- 审计留痕
- 质量告警
- 调度协同

### 10.6 失败、降级与重试路径

系统需要清晰区分三类路径：

- **失败**：无可用 Provider、强约束冲突、持久化异常等
- **降级**：允许使用次优 Provider 或 degraded 入库
- **重试**：仅在策略允许时对 Provider 拉取执行有限重试

## 11. 路由与调度机制

### 11.1 Provider 注册机制

建议通过 `ProviderRegistry` 维护所有 Provider 的统一注册信息，包含能力、市场、优先级和可选健康信息。

### 11.2 能力匹配机制

第一层过滤基于请求能力与 Provider 能力集合进行匹配，不满足能力要求的 Provider 直接排除。

### 11.3 市场与数据域匹配机制

第二层基于市场和数据域匹配，确保请求不会落到语义不符的数据源上。

### 11.4 优先级与排序策略

候选 Provider 在满足能力与市场要求后，应按优先级排序。后续可引入更多排序因子，但骨架阶段不强制实现。

### 11.5 降级与回退策略

骨架阶段保留最小降级策略：

1. 能力过滤
2. 市场过滤
3. Provider 强制约束过滤
4. 优先级排序
5. 失败后降级到下一个候选

### 11.6 熔断与限流预留位

以下机制保留扩展点但暂不作为骨架阶段必选项：

- 熔断器
- 配额惩罚
- 健康探针结果排序
- 动态 Provider Policy

## 12. 数据标准化与质量治理设计

### 12.1 标准化模型

系统对外应输出统一数据结构，对内允许不同 Provider 保留各自适配逻辑，但适配结果必须收敛到统一标准模型。

### 12.2 字段映射规范

字段映射至少需要覆盖：

- 时间字段
- 价格字段
- 成交量字段
- symbol / market / dataset 元信息字段

### 12.3 数据清洗规则

骨架阶段要求支持以下基础清洗能力：

- 空值与空白值治理
- 重复记录清理
- 数值与时间类型标准化
- 关键字段缺失识别
- 元信息补齐

### 12.4 DQ 评分与决策口径

DQ 评分规则不应散落在业务逻辑中，而应由独立 DQGate 配置与质量引擎统一决定。阈值必须配置化，而非硬编码到上游调用代码中。

### 12.5 Reject / Degraded / Accept 规则

推荐的统一口径如下：

- `score >= accept_threshold`：正常入库
- `degrade_threshold <= score < accept_threshold`：降级入库，并发 degraded 事件
- `score < degrade_threshold`：拒绝入库，写 quarantine，并发 rejection 事件

### 12.6 Quarantine 机制

reject 数据不得直接丢弃，而应写入 quarantine，并保留至少以下信息：

- dataset_id
- 来源 Provider
- 质量分数
- 发现的问题列表
- 行数与清洗后行数
- 时间戳
- 相关列信息
- 元数据快照引用

---

## 13. 存储与状态管理设计

### 13.1 主存储设计

骨架阶段的最小要求是能够将清洗后的数据写入统一数据平面存储，例如 Parquet 或其等价实现。

### 13.2 元数据与目录设计

系统必须在主存储写入后同步写入 catalog / metadata，以支持：

- 数据发现
- 审计与追踪
- 质量报告回写
- 后续下游读取

### 13.3 PiT 语义

PiT、available_at、版本时间等高级时间语义在骨架阶段可只保留接口与元数据位，不要求一次性做完整实现。

### 13.4 状态投影与幂等要求

统一数据管道需要具备最小幂等能力，避免重复写入和重复发射关键事件。状态投影应当可与当前控制面保持一致。

### 13.5 缓存设计预留位

Redis 或其他缓存体系不作为当前阶段前置依赖，但应在接口层预留未来接入空间，避免后续侵入式改造。

---

## 14. 事件与异步集成设计

### 14.1 事件模型

事件模型应围绕数据拉取、质量门控、摄入完成、拒绝、降级等核心动作设计，保持命名稳定、字段稳定、版本清晰。

### 14.2 事件发布策略

本阶段优先复用当前事件机制或 records writer 风格接口，避免直接将 Kafka 作为主链路硬依赖。

### 14.3 与 Job / Agent / EventBus 的集成方式

建议采用以下集成模式：

- `fetch_dataset()` 与 `ingest_dataset()` 暴露为 Job Handler 能力
- 长耗时或多步骤任务可由 Agent 编排推进
- 事件通过 EventBus 广播给质量告警、审计或其他消费者

### 14.4 Kafka 替换扩展点

Kafka 在后续阶段可承接更强的异步事件能力，但应当是对底层发布机制的替换，而非对上层语义的改写。

### 14.5 下游消费者扩展点

下游可逐步扩展：

- 数据质量告警消费者
- Catalog 同步消费者
- 指标聚合消费者
- 工作台推送消费者
- 审计归档消费者

---

## 15. 可观测性与审计设计

### 15.1 日志规范

系统至少应输出以下日志维度：

- provider_name
- dataset_id
- capability
- market
- request_id / trace_id
- latency
- fallback_count
- decision
- quality_score

### 15.2 指标体系

最小指标建议包括：

- fetch 次数与成功率
- ingest 次数与成功率
- provider 维度延迟
- fallback 次数
- reject / degraded / accept 计数
- quarantine 写入次数

### 15.3 Trace 设计

每次 fetch / ingest 请求应进入 tracing span，便于串联：

- 请求入口
- Provider 解析
- 拉取执行
- 标准化
- DQ
- 持久化
- 事件发射

### 15.4 审计事件设计

关键动作应具备审计留痕，包括：

- 谁发起请求
- 请求了什么数据
- 使用了哪个 Provider
- 质量决策为何
- 是否落入 quarantine
- 最终写入何处

### 15.5 故障定位与排障视图

通过日志、指标、trace 与事件组合，系统应支持快速回答以下问题：

- 本次数据为何被 reject
- 本次为何走了 fallback
- 数据落盘是否成功
- 元数据是否已写入
- 是否已发射质量事件

---

## 16. 安全、权限与合规要求

### 16.1 凭证管理

Provider 访问凭证不得散落在业务代码中，应统一通过配置与安全存储方式注入。

### 16.2 Provider 访问控制

不同 Provider 可根据环境、用途或账号权限进行访问限制，禁止未授权调用。

### 16.3 数据分级与脱敏要求

对于敏感数据集，应预留数据分级与脱敏治理边界，即便当前阶段不做完整实现，也应避免后续接口失配。

### 16.4 审计留痕要求

涉及数据获取、质量拒绝、隔离区写入、重要元数据更新等关键动作，应具备最小审计留痕能力。

---

## 17. 非功能需求

### 17.1 性能目标

系统应保证在骨架阶段具备可用的单机性能，满足当前项目规模下的数据拉取与摄入闭环需求。

### 17.2 可用性目标

在某个 Provider 失败时，系统应能在策略允许范围内自动降级或显式失败，而非静默挂起。

### 17.3 一致性目标

上层契约、质量门控口径、元数据回写和事件语义必须一致，避免同一数据集在不同链路中出现不同定义。

### 17.4 可扩展性目标

系统必须支持后续逐步接入更多 Provider、缓存、异步总线与控制台能力，而不破坏现有接口。

### 17.5 可维护性目标

模块边界应明确，核心抽象清晰，便于测试、替换与后续团队协作维护。

---

## 18. 实施分阶段规划

### 18.1 Phase A：骨架落地

交付内容：

- `DataPipelineManager`
- ProviderContract
- ProviderRegistry
- RouteResolver（最小版）
- Normalizer
- DQGate 接口
- PersistWriter 接口
- EventEmitter 接口

### 18.2 Phase B：路由与 DQ 强化

交付内容：

- 至少 1~2 个基础 Provider
- `fetch_dataset()` 可运行
- `ingest_dataset()` 可运行
- reject / degraded / accept 三态闭环
- quarantine 可写
- metadata / catalog 可写

### 18.3 Phase C：异步化与可观测增强

交付内容：

- 作为 Job Handler 接入
- 与当前事件总线衔接
- tracing / metrics 接通
- 基础回归测试建立

### 18.4 Phase D：缓存、Kafka、健康检查等增强能力

后续按需启用：

- Redis 热缓存
- Kafka 审计事件
- 健康探针
- 配额感知路由
- WebSocket Bridge
- 更多 Provider

## 19. 风险、约束与待决事项

### 19.1 已知风险

- Provider 契约定义过弱会导致后续扩展反复返工
- DQ 若未真正进入主链路，后续 reject / degraded 语义会失真
- 持久化与事件发射若边界不清，审计与排障成本会升高
- 过早引入过多增强能力会稀释骨架阶段目标

### 19.2 当前约束

- 当前阶段必须优先保证骨架闭环，而非全量功能覆盖
- 统一入口是强约束，不应允许上游长期绕行
- 阈值、策略、降级规则需尽量配置化
- 需要兼容现有 Job / Agent / EDA 基座

### 19.3 待决问题清单

- `DataPipelineManager` 与 `DataSourceManager` 最终命名是否统一
- 首批 Provider 的选择范围与优先级口径
- Metadata / catalog 的最终存储结构
- DQ 报告对外暴露的最小字段集合
- 后续 Kafka 事件命名是否直接承接当前领域事件结构

### 19.4 风险缓解建议

- 先锁定统一契约，再推进更多 Provider
- 先建立最小 ingest 闭环，再做复杂异步增强
- 对 reject / degraded / accept 三态建立集成测试
- 所有扩展点以“替换实现”而非“新增平行主链路”为原则

---

## 20. 验收标准

### 20.1 功能验收

- 存在唯一统一入口
- 至少支持 `fetch_dataset()` 与 `ingest_dataset()`
- 至少支持一个 market / reference / mock provider 组合
- 可以完成正常入库、降级入库、拒绝入库三种路径
- 可以写 quarantine 与 metadata

### 20.2 契约验收

- Provider 通过统一契约接入
- 调用方不再硬编码底层第三方源
- Capability / Market / Dataset 契约清晰稳定
- 事件字段语义清晰且版本明确

### 20.3 质量验收

- DQ 进入 ingest 主链路
- reject / degraded / accept 规则明确
- 质量报告可随结果返回并回写元数据
- quarantine 中可追溯 reject 原因

### 20.4 可观测性验收

- 具备 tracing / metrics / structured logs
- 能定位 Provider 拉取、标准化、DQ、持久化、事件发射各阶段问题
- 能统计 fallback、reject、degraded 与 success

### 20.5 扩展性验收

- 后续可平滑引入 Redis / Kafka
- 后续可扩展更多 Provider
- 不破坏上层调用接口
- 可与 Job / Agent / EDA 体系自然衔接

## 结论

本合并正式版文档的核心结论是：当前阶段最适合建设的是统一数据管道的“骨架”，而不是旧系统的完整运行形态。新项目应当保留统一入口、契约驱动、能力路由、标准化、质量门控、持久化、事件发射与可观测这些稳定核心，并以现有 Phase 1 EDA 内核作为控制面与执行面基座，逐步向更完整的数据平台能力演进。
